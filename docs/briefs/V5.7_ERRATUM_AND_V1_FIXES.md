## Part 1 — Specification erratum (no code changes; the code is already right)

### 1.1 §6 tiebreak order — corrected to the Spielordnung

v5.6 §6 gave a tiebreak order sourced from a third party. The §11 verification
against the DFL Spielordnung (SpOL, Stand 2026-03-06, § 5 Nr. 3 c) found it
wrong, and the engine correctly follows the primary source. The specification
now reads:

Once clubs are level on points:
1. Tordifferenz (Subtraktionsverfahren)
2. Anzahl der erzielten Tore
3. Gesamtergebnis aus Hin- und Rückspiel im direkten Vergleich
4. Anzahl der auswärts erzielten Tore im direkten Vergleich
5. Anzahl aller auswärts erzielten Tore
6. Entscheidungsspiel auf neutralem Platz

There is **no "Punkte im direkten Vergleich" step** — v5.6's chain contained
one and it does not exist. For three or more level clubs, criteria 3–4 operate
on a **mini-table over only the tied clubs' mutual matches, recomputed after
every partial separation**; this is a documented interpretation (the SpOL
wording is written for the two-club case), recorded in
`docs/verification/dfl-spielordnung.md` §4.4.

**In-season rules (SpOL, previously missing from the brief entirely):**
- Before a tied group has played **both legs** among themselves, only criteria
  1–2 apply; anything still level shares a table position
  (**geteilter Tabellenplatz**).
- Once both legs are played, criteria 3–5 apply in-season too.
- Criterion 6 **never** applies during a running season.

**Derived requirements, binding for the UI and every consumer:**
- Tabelle & Prognose renders shared positions as shared (same rank number,
  visually marked), never inventing an order.
- Zone tallies, clinch logic and every feature needing "the position" consume
  the **position range** (`positionRange`), not a single rank.
- The §6/§7 conservative clinch rule stands unchanged — it never relied on the
  wrong chain, and under the corrected chain it is, if anything, safer.

## Part 2 — Engine corrections (DO FIRST — cheap only until 2026/27 artefacts exist)

The 2025/26 artefacts are retrospective and regenerable in one pipeline run.
Once the 2026/27 season starts committing artefacts every matchday, any change
below multiplies in cost. All four land in **one commit** with **one** bump of
`SIMULATION_PROTOCOL_VERSION` (1 → 2) and a regeneration of every committed
artefact under `data/seasons/`.

### 2.1 Decider draw gets a proper random key

`simulateSeason` currently draws the criterion-6 stand-in as
`uniform01(noiseKey[club], run ^ 0x5f5f5f5f)` — reusing the noise key with a
mangled run index. That violates the documented key schema ("every draw is a
pure function of its key") and hides a convention in an XOR constant. Replace
with a dedicated key:

```js
const deciderKey = clubs.map((c) =>
  makeKeyBase({ seasonId, context, id: c.clubId, drawKind: "decider" }));
// ...
decider: (clubId) => uniform01(deciderKey[idx.get(clubId)], run)
```

### 2.2 KeyBase uniqueness assertion

`makeKeyBase` folds to 32 bits; ~500 keys per season give a ~3·10⁻⁵ collision
probability, and a collision silently couples two fixtures to one stream.
`simulateSeason` asserts at setup that all keyBases (noise, fixture, decider)
are pairwise distinct and throws otherwise. One `Set`, one check.

### 2.3 Guard against half-defined played fixtures

A fixture with exactly one of `gh`/`ga` defined is currently classified as
unplayed and silently resimulated, discarding the real half-result. Throw
instead: `fixture ${id} has gh xor ga — refusing to guess`.

### 2.4 Backfill politeness delay

The one-time history backfill fetches sequentially (good) but with no pause.
Insert a fixed delay (750 ms) between clubelo requests in the backfill path.
The 2-hour cron (one daily CSV) needs nothing. Development and pipeline tests
must never loop against the live API — recorded fixtures only (the engine
tests already comply; this makes it binding for the pipeline too).

### 2.5 Archive storage adapter

Snapshot archiving writes through a single storage module whose base location
comes from configuration (default: `data/ratings/` in this repo). Moving the
archive to a private repo must be a configuration change plus a migration
commit — never a refactoring. Path semantics (`index.json`, snapshot naming,
idempotent atomic append) are location-independent.

**Regeneration note:** after 2.1–2.3, regenerate `data/seasons/2025/**` in the
same commit; the artefacts record `simulationProtocolVersion: 2`. Expected
diff: probabilities move within Monte-Carlo noise only (the decider fires
rarely); assert per-target |Δ| < 3·SE using the stored batch frequencies, and
abort if exceeded — that would indicate a real behaviour change, not the key
cleanup.
